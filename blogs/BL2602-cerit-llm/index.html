Claude Code na české akademické infrastruktuře: poctivý benchmark CERIT LLM

Michal Cifra · Ústav fotoniky a elektroniky AV ČR · Duben–červen 2026 · ~2 200 slov

CERIT-SC z e‑INFRA CZ provozuje veřejně přístupnou LLM gateway (llm.ai.e-infra.cz) s Anthropic-kompatibilním rozhraním — což znamená, že Claude Code lze přesměrovat na CERIT modely jediným přepsáním jedné proměnné prostředí. Provedl jsem přes 600 agentických běhů na 10 reálných vývojářských úlohách a porovnal, co tahle free infrastruktura reálně zvládne oproti nativní Anthropic službě. Výsledek: Gemma 4 dosahuje 91 % výkonu Claude Sonnet — s nulovými náklady a konkrétními rozdíly, které stojí za to pochopit dřív, než přepnete tým.

Jak benchmark funguje

Benchmark je postaven přímo na úlohách, které dělám každodenně: psaní agentů (Telegram, polling, systemd unit), datové pipeline (RSS, TF-IDF), infra skripty, LLM integrace, REST klienti, zpracování PDF, vector DB + RAG, orientace v kódu, konfigurace prostředí a social-media autopostery. Každá úloha je spouštěna v čistém sandboxu (/tmp/bench-XXXX) s formálním zadáním, sadou očekávaných výstupních souborů a pytest testy.

Čtyři tracky = čtyři způsoby volání

TrackPopisPočet trialů
A Nativní Claude Code (claude --print, Anthropic API) 100
B CERIT modely přes OpenAI-shape API + vlastní Python harness 30
C CERIT přes Claude Code + LiteLLM proxy (překlad protokolu) 10
D CERIT přes Claude Code přímo na /v1/messages (Anthropic-shape) 20–100

Track A a D jsou metodologicky srovnatelné — oba používají Claude Code s kompletní sadou nástrojů (Bash, Read, Write, Edit, Glob, Grep, TodoWrite). Track B testuje „model samotný" přes vlastní Python harness. Track C přidává LiteLLM vrstvu překladu, která v praxi způsobuje problémy (viz níže).

Čtyřvrstvá evaluace a composite score

Každý běh projde čtyřmi vrstvami hodnocení, které se slučují do jednoho composite skóre na škále 0–1: 

composite = 0.40 × correctness + 0.25 × completeness
          + 0.20 × quality   + 0.15 × efficiency

correctness = 0.20 × syntax + 0.40 × pytest_tests + 0.40 × LLM_judge
efficiency  = min(1.0, 15 000 / total_tokens)
Metodická poznámka: jako LLM judge je použit model z rodiny Claude — ten samý dodavatel jako Track A. To může systematicky zvýhodňovat Anthropic modely ve složce judge (40 % z correctness, tj. 16 % celkového skóre). Složka pytest_tests (40 % z correctness) je na tomto zkreslení nezávislá a je proto metodicky robustnější.

Co stojí za čísly: dimenze skóre

Composite je vážený průměr čtyř dimenzí. Každá z nich říká něco jiného. Níže jsou tři klíčové modely:

Model Correctness Completeness Quality Efficiency Composite

gemma4 (Track D) 0.8821.000 0.6670.902 0.807

Tři důležité pozorování:

1. CERIT modely mají vyšší correctness než Sonnet. Gemma4 (0.882) a kimi-k2.5 (0.892) překonávají Claude Sonnet (0.797). Correctness zahrnuje z 40 % pytest testy — ty jsou nezávislé na LLM a metodologicky nejrobustnější část skóre. CERIT modely tedy v průměru projdou více automatizovaných testů.

2. Claude vede na quality a efficiency. Quality (lint, bezpečnostní vzory, ošetření chyb) je u Sonnet 0.944 oproti 0.667 u CERIT modelů. Kód z nativního Claude je čistší a bezpečnější. Efficiency odráží tokenovou hospodárnost: Claude produkuje výsledky za ~4 000 tokenů (úspěšné běhy), CERIT Track D potřebuje 17 000–45 000.

3. Rozdíl composite je +0.034 ve prospěch Sonnet. Gemma4 dosahuje 0.807 oproti Sonnetu 0.841 — rozdíl je podstatně menší než ~0.10, které naznačoval dřívější, menší vzorek (30 trialů, méně modelů). Po rozšíření benchmarku a přidání gemma4 se gap výrazně zmenšil.

Spolehlivost: konvergence má svůj příběh

Konvergence znamená, že Claude Code subprocess skončil s returncode == 0. Toto nezaručuje správnost výstupu, jen že se model vůbec dočkal závěru bez timeoutu nebo fatální chyby. I přes toto omezení je číslo operačně relevantní: nekonvergenující běh neprodukuje žádné soubory.

Nižší konvergence u Claude (50–62 %) v plném benchmarku (100 trialů, složitější task varianty) oproti dřívějšímu běhu (93 %) má konkrétní příčinu: Sonnet typicky provede 4 tool cally — na jednoduché úlohy to stačí, na zadání vyžadující 4+ souborů (Dockerfile + compose.yaml + .env + inicializační skript) nestačí a výstup je neúplný. CERIT modely s 10–13 tool cally mají vyšší šanci pokrýt vše.

Klíčový závěr: Claude generuje čistší, bezpečnější kód v méně krocích. CERIT modely (zejména gemma4, kimi-k2.6) jsou méně elegantní, ale spolehlivěji dotáhnou úlohu do konce. V produkčním agentickém prostředí (unattended provoz) má spolehlivost stejnou váhu jako kvalita.

Na první pohled dramatické: CERIT modely výrazně překonávají Claude na úlohách 07–10. Ale za tím číslem stojí konvergence: Claude Sonnet selže (0 souborů, subprocess error) ve ~50 % trialů a těchto úlohách zejména. CERIT modely s více tool cally tyto úlohy dotahují. Kdy se Sonnet skutečně podaří (converge), skóre jsou srovnatelná.

Výjimkou je úloha 07 (Vector DB + RAG), kde propadají i Opus (0.398) — zde jde o skutečnou obtížnost úlohy, která stojí na kombinaci Qdrant + embedding pipeline + multi-step přípravy dat. CERIT modely ji zvládají spolehlivěji díky větší ochotě k iterativnímu opravování přes více tool callů.

Tokenová efektivita: čísla bez iluzí

Nativní Claude potřebuje na úspěšný běh ~4 000 tokenů a 4–8 tool callů. CERIT modely v Track D potřebují 17 000–45 000 tokenů a 10–14 tool callů. Rozdíl je reálně 4–10×, ne řádový.

Větší poměry (až 70× a více) se vyskytují v Track B (vlastní Python harness), kde modely jako DeepSeek nebo agentic alias spotřebují 230 000–540 000 tokenů na úlohu. To je artifact toho, jak harness posílá dotazy — nikoli vlastnost modelu.

Pro CERIT uživatele: větší spotřeba tokenů není problém, protože CERIT neúčtuje per-token. Provozní cena je nulová bez ohledu na to, zda model potřebuje 5 000 nebo 50 000 tokenů.

Nalezená chyba: thinker a deepseek-v3.2-thinking

Modely thinker (alias) a deepseek-v3.2-thinking přes Anthropic-shape rozhraní (/v1/messages) vykazují strukturální chybu: nativní DeepSeek tokeny <｜DSML｜function_calls> a <｜DSML｜invoke name="…"> prosakují do obsahu bloku thinking místo toho, aby byly serializovány do struktury tool_use. Claude Code vidí model jako přemýšlení a tool call nikdy nevyvolá. Výsledkem jsou session bez jediného vytvořeného souboru.

Stejné modely přes Track B (OpenAI-shape API) fungují bez problémů — composite 0.728, medián 15 reálných tool callů na trial. Chyba je tedy specifická pro Anthropic-shape adaptér. Bylo provedeno 165 izolovaných curl volání bez reprodukce chyby v minimálním nastavení — spouštěč pravděpodobně spočívá v kombinaci 23K-tokenového systémového promptu Claude Code a 30+ nástrojů. Hlášení s reprodukcí a kvantitativním dopadem bylo zasláno správci CERIT služby Lukáši Hejtmánkovi dne 2026-04-25. Oba modely jsou z analýzy vynechány.

Praktické doporučení: Vyhněte se v tuto chvíli modelům thinker a deepseek-v3.2-thinking přes Anthropic-shape rozhraní. Přes OpenAI-shape fungují — ale to vyžaduje vlastní harness (Track B), ne Claude Code CLI.

Praktická konfigurace: switcher pro tým

Switcher je implementován jako sada shell funkcí v ~/.bashrc. Výběr modelů se opírá o benchmark výsledky; middleware vrstva (rewrite proxy na localhost:9999) navíc odemkne živé UI Claude Code (spinner, token countdown), které jinak vyžaduje model s názvem odpovídajícím claude-*. 

claude-cerit() {
  ANTHROPIC_BASE_URL="https://llm.ai.e-infra.cz/"  \
  ANTHROPIC_MODEL="gemma4"                          \  # ← vítěz Track D
  ANTHROPIC_DEFAULT_OPUS_MODEL="gemma4"             \
  ANTHROPIC_DEFAULT_SONNET_MODEL="gemma4"           \
  ANTHROPIC_DEFAULT_HAIKU_MODEL="glm-5"             \  # ← rychlý, 45 s median
  MAX_THINKING_TOKENS=0                             \
  DISABLE_TELEMETRY=1                               \
  command claude "$@"
}

claude-cerit-rich() {        # s živým UI přes proxy
  # viz cerit-bashrc.snippet v ~/dev/ pro plnou implementaci
  command claude --model claude-sonnet-4-6 "$@"    # proxy přepíše na gemma4
}

Pozor na stabilitiu aliasů: případ thinker ukázal, že CERIT může přemapovat alias na jiný backend se zásadně odlišnou paritou kvality, aniž by to bylo vidět z uživatelské perspektivy. Pro produkční provoz jsou konkrétní pojmenované modely (gemma4, kimi-k2.6) bezpečnější než aliasy.

Závěr

Výsledek po 600+ bězích na 10 reálných úlohách je konkrétní: nejlepší CERIT model (gemma4) dosahuje composite skóre 0.807 oproti 0.841 Claude Sonnet — to je 96% shoda na úrovni kompozitu. Na dimenzích se liší: Claude vede na čistotě kódu a quality heuristikách, gemma4 a kimi-k2.6 vedou na correctness (pytest testy) a spolehlivosti dokončení.

Pro týmy, kde ne každý člen má předplacenou Anthropic subskripci, je CERIT reálná alternativa pro 80 % rutinní agentické práce — psaní testů, doplňování kódu, refactoring, orientace v repozitáři. Na architekturální diskuse nebo složité multi-session refaktory doporučuji nativní Claude.

Poděkování: Velký dík CERIT-SC za provoz LLM gateway a otevřený přístup pro českou akademii. Specifický dík Lukáši Hejtmánkovi za rychlou reakci na hlášení chyby a nabídku spolupráce na reprodukci.
