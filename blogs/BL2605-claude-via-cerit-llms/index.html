Claude Code on Czech academic AI — the full story

A local proxy routes Claude Code’s agentic tool-calls to the free CERIT LLM gateway. This post covers everything: what broke, why, six proxy interventions, a 117-task benchmark, and the idle-stop mystery that turned out to be more complicated than expected.

Michal Cifra · June 2026 · github.com/mcer33/claude-code-cerit-llm
TL;DR
  • Czech academics get free LLM API access at llm.ai.e-infra.cz (MetaCentrum / MUNI account required)
  • Naive forwarding fails in 3 ways; a ~700-line local proxy fixes all of them
  • Best model: GLM-5.2 (thinking disabled) — 494 s · 7/7 · 0 idle stops · 0 errors · 0 context inflation
  • Extended 117-test suite: 88% completion · 0 idle stops · 947 tool calls · avg quality 5.37/10
  • The “idle-stop problem” is real in complex multi-turn sessions but context-dependent — simple calls work fine
  • Code + benchmark suite: github.com/mcer33/claude-code-cerit-llm

1. The opportunity

CERIT-SC, part of the e-INFRA CZ national research infrastructure, runs a free LLM gateway at llm.ai.e-infra.cz — an Anthropic-compatible /v1/messages endpoint, no usage fees for MetaCentrum or MUNI members. For researchers already using MetaCentrum HPC, it is a natural complement: the same account covers both GPU compute and LLM access.

The case is not only cost. Data stays on Czech national infrastructure — relevant for pre-publication research code. No cloud vendor, no usage cap, no billing alarm. And because the endpoint exposes the full Anthropic messages API shape, Claude Code can talk to it with minimal friction once a small compatibility proxy is in place.

2. Three problems with naive forwarding

The naive approach: set ANTHROPIC_BASE_URL=https://llm.ai.e-infra.cz and launch Claude Code. Three things break immediately.

Problem 1 — Tool-definition rejection (HTTP 400)

Claude Code sends Anthropic proprietary tool types: web_search_20250305, computer_20241022, text_editor_20250429. CERIT’s vLLM / sglang backends reject these with HTTP 400. Claude Code crashes before accomplishing anything. No workaround on the client side; the proxy must intercept and sanitize every request before forwarding.

Problem 2 — GLM-5.2 thinking mode

GLM-5.2 has thinking mode on by default. Without disabling it, the model burns 143 tool calls thinking about the first task and times out at 900 seconds with no useful output. Disabling via chat_template_kwargs: {"enable_thinking": false} brings T1 wall time from 900 s to 48 s. This single intervention accounts for most of GLM-5.2’s benchmark advantage.

Problem 3 — Model name / TUI spinner

Claude Code only shows its live progress spinner when ANTHROPIC_MODEL starts with claude-. With glm-5.2 or kimi as the model name, the TUI looks like a hang with no output. The proxy preserves a spoofed claude- model name toward Claude Code while sending the real model name upstream.

Additional bug — DeepSeek thinking tokens

With deepseek-v4-pro-thinking, internal chain-of-thought tokens leaked into the text response, corrupting tool-call JSON. Filed as ticket #1519943 (April 2026). A separate server-side issue from the three core problems above.

3. The proxy — six interventions

The proxy runs locally as a ThreadingHTTPServer on 127.0.0.1:9999, intercepts all /v1/messages requests from Claude Code, transforms them for CERIT compatibility, and relays SSE streams back. About 700 lines of Python, no external dependencies beyond the standard library. Here are the six interventions it applies.

1 · Tool sanitizer

Strips or converts Anthropic proprietary tool types (web_search_20250305, computer_20241022, text_editor_20250429). Patches missing input_schema fields. Strips spurious outer quotes from tool-call JSON — a gemma4-specific formatting bug where the model wraps its entire JSON output in an extra string layer.

2 · Continuation injection

Appends CERIT_CONTINUATION to the system prompt. The model signals task completion with TASK_COMPLETE on its own line. Also injects tool_choice: {type: any} to force a tool call rather than a prose response. Note: this was tested on kimi and caused a thinking-phase deadlock — removed for kimi sessions.

CERIT_CONTINUATION system prompt injection 
AGENTIC EXECUTION — NON-NEGOTIABLE
When tools are available: call a tool immediately. Do not explain. Do not narrate.
FORBIDDEN before a tool call: 'I will now' / 'Let me' / 'I'll first'
end_turn is ONLY valid: (a) task 100% done — write your FULL final answer now,
  OR (b) you need information only the human can provide.
All other turns: CALL A TOOL. No exceptions.
Completion signal: when the task is 100% done, end your final response with: TASK_COMPLETE

3 · Turn + repetition guards

At turn 12, a synthesis nudge is injected. At turn 20, a hard stop message is sent. If any single tool has been called 3 or more times in the last 30 messages, a loop-break injection fires. These prevent runaway sessions that accumulate context without making progress.

4 · GLM thinking disable

For any request targeting glm-5.2, the proxy injects chat_template_kwargs: {"enable_thinking": false} into the request body before forwarding. Wall time for T1: 900 s → 48 s. The most impactful single intervention in the entire proxy.

5 · Fallback chain

On HTTP 400 context overflow: glm-5.2 → kimi → qwen3.5-122b. Each model in the chain has progressively larger context windows (110K → 131K → 256K), covering most real-world session lengths.

6 · HTTP 429 retry

Three retries with 5-second backoff on HTTP 429 rate-limit responses. A thread-safe REQUEST_COUNTER logs all upstream requests for debugging. During run10, GLM-5.2 itself implemented this counter — the benchmark doubled as a live coding session.

Two issues discovered: (1) kimi underperforms as tool target; (2) the idle-stop metric was double-counting. The stream-JSON emitter fires 2 events per message (text + tool_use), both counted as separate turns — and the final text-only turn always counted as idle. Both bugs fixed before run6.

Faster than qwen3.5-122b, but with 68 upstream errors and 2.2–2.4× context inflation. 188 continuation injections required. Technically functional but operationally unreliable.

Thinking mode enabled (default). T1 alone burned 143 tool calls and hit the 900-second wall. Fix: add the thinking-disable intervention → run9.

5. The idle-stop mystery

This deserves its own section because it is the most interesting part of the investigation — and the answer is messier than we first thought.

What we observed

In complex Claude Code sessions — 20+ tool definitions, multi-turn, long context — the model would output text describing what it would do instead of calling a tool. stop_reason: end_turn. Claude Code interprets this as session completion and exits. Task incomplete.

What we measured

First tests showed up to 37% idle-stop rate. We built the continuation-injection proxy to fight it. The proxy tracks every turn, classifies it as tool-call or idle, and injects CERIT_CONTINUATION on the next message if the model idles.

What we sent to the admin

Three support requests, including asking for --enable-auto-tool-choice on kimi — assuming a vllm backend. We were confident this was a server-side configuration issue.

What the admin said

Admin Lukas Hejtmanek (CERIT-SC): all models have tool parsers configured. kimi runs on sglang, not vllm — the flag does not apply. GLM-5.2 is far better than the GLM-4.x we had been referencing. The difference is significant.

What the reproducer shows today

A minimal 1-tool, 1-message request to both qwen3.5-122b and kimi → 0% idle-stop, no continuation needed. Simple calls work fine.

Resolution

Finding

The idle-stop is context-complexity dependent. Simple calls work fine. Complex multi-turn sessions with the full Claude Code tool suite (~20 tools, long message history) are where it manifests.

The 37% figure from night 1 was also inflated by a metric bug (double-counting, fixed before run6). The continuation prompt in the proxy still helps in real complex sessions — run10 recorded 0 idle stops across 947 tool calls.

Whether the LiteLLM middleware layer (broken for Claude Code since v1.81.14, confirmed by admin, not yet fixed) plays a role in complex-session idle-stops remains an open question.

6. What the admin confirmed

Admin Lukas Hejtmanek (CERIT-SC), confirmed June 19, 2026:

  • All deployed models have tool calling and parsers configured server-side
  • qwen3-coder = alias for qwen3.5-122b (same model, same weights)
  • Use kimi alias — tracks current version (k2.7 as of June 2026); never pin to numbered suffix
  • GLM-5.2 is far better than old GLM-4.x (confirmed ≈ Opus 4.8 per MCP Atlas)
  • LiteLLM frontend: broken for Claude Code since v1.81.14, not yet fixed
Key lesson

Check /v1/models first. The full list of available models is always there and avoids redundant admin requests. New models appear without announcement. We wasted two support cycles asking about models that were already deployed.

7. Getting started

Setup — four steps 
# 1. Get a MetaCentrum or MUNI account → llm.ai.e-infra.cz → token
mkdir -p ~/.config/cerit
echo 'YOUR_TOKEN' > ~/.config/cerit/token && chmod 600 ~/.config/cerit/token

# 2. Clone + install
git clone https://github.com/mcer33/claude-code-cerit-llm ~/dev/claude-code-cerit-llm
cd ~/dev/claude-code-cerit-llm && bash install.sh

# 3. Verify
source ~/.bashrc && claude-cerit-ping

# 4. Launch
claude-cerit-rich    # GLM-5.2, 110K context — recommended default

8. What’s next / open questions

  • L01 test: llama-4-scout tool verification pending. Long-context scenarios need a proper structured benchmark, not just anecdotal testing.
  • DeepSeek V4 Pro re-test: Admin is working on sglang migration for DeepSeek — vllm currently has compilation issues for some workloads. Once migrated, re-run the core 7-task suite to see if error rate and context inflation improve.
  • LiteLLM fix: Broken for Claude Code since v1.81.14 (CERIT-side, not yet fixed). This may be the root cause of complex-session idle-stops. Worth a targeted test once resolved.
  • Idle-stop in complex sessions: A more realistic reproducer — 20+ tools, multi-turn, full production context — would better characterise the remaining phenomenon versus the simplified 1-tool reproducer that shows 0% idle.
Bottom line

For routine agentic coding work on research projects, GLM-5.2 with thinking disabled is the recommendation. It is fast, clean, and requires no babysitting. For sessions that exceed 80K tokens, fall back to qwen3.5-122b. Both are free, both run on Czech national infrastructure, and both outperform expectations for open-weight models.
