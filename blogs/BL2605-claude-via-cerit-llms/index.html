Claude Code on Czech academic AI — free agentic coding with GLM-5.2

A 620-line Python proxy routes Claude Code's tool-calling agent to the free CERIT LLM gateway at llm.ai.e-infra.cz. After benchmarking four models across seven real research-engineering tasks, GLM-5.2 with thinking disabled emerges as the winner: 7/7 tasks, 494 s total, 0% idle-stop, zero context inflation — at zero token cost.

📅 June 2026 · 📋 Michal Cifra, UFE CAS · 🔗 GitHub

Why bother?

Claude Code is Anthropic's agentic CLI: it reads your codebase, calls shell tools, writes files, and iterates — often completing in minutes what would take an hour manually. For heavy users, the API bill is $25–100+ per month. Czech academic researchers, however, already have access to a free frontier-model gateway through the e-INFRA CZ national research infrastructure, operated by CERIT-SC at Brno. The gateway at https://llm.ai.e-infra.cz exposes an OpenAI-compatible API — and, crucially, an Anthropic-compatible endpoint — running on NVIDIA DGX H100/B200 hardware. Models include GLM-5.2 (≈ Claude Opus 4.8 per independent evals), qwen3.5-122b, DeepSeek V4 Pro (1 M token context), and kimi-k2.6, all available at no token cost to MetaCentrum users.

The motivation is not only cost. Running on academic infrastructure means data stays on Czech national infrastructure, which matters for pre-publication research code. And because the endpoint exposes the full Anthropic messages API shape, Claude Code can talk to it with minimal friction — once a small compatibility proxy is in place.

Who this is for: MetaCentrum / MUNI / e-INFRA CZ users who want to run Claude Code agent sessions on their own research machines (Linux workstation, university server, or a remote JupyterHub node) without an Anthropic subscription. The proxy runs locally; your CERIT API token never leaves your machine.

The challenge: three things Claude Code expects that CERIT doesn't provide

Pointing ANTHROPIC_BASE_URL at llm.ai.e-infra.cz and setting a CERIT token gets you a working chat session. But the agent loop — where Claude Code reads files, calls tools, and iterates — breaks in three distinct ways:

1 — Idle-stop: the model outputs text instead of calling a tool

Claude Code's agent loop expects the model to call at least one tool per turn until the task is done. When the model instead produces a plain-text turn with finish_reason: end_turn and no tool_use block, Claude Code treats the session as complete and exits — even if the task is half-finished. In early benchmarks with the proxy naively forwarding requests, 33% of turns were "idle stops." For reference, with native Claude Sonnet the rate is near zero.

2 — Runaway sessions: the model loops without converging

The opposite problem: some models, particularly GLM-5.2 with its default reasoning mode active, call tools in bursts of 9–10 per turn and never synthesize a final answer. One T1 run accumulated 143 tool calls in 15 turns and timed out at 900 seconds without producing output.

3 — The TUI spinner gates on model name

Claude Code's live progress UI (token countdown, spinner) is shown only when ANTHROPIC_MODEL starts with claude-. Setting it to gemma4 or glm-5.2 produces a degraded, silent interface that looks like a hang.

Solution: a local rewrite proxy

The full solution is a single Python file (cerit-rewrite-proxy.py, ~620 lines) that runs as a local HTTP server on 127.0.0.1:9999. Claude Code talks to the proxy as if it were Anthropic; the proxy forwards to CERIT with appropriate rewrites.

Claude Code ──→ proxy :9999 ──→ llm.ai.e-infra.cz
ENV: ANTHROPIC_BASE_URL=http://127.0.0.1:9999/
ENV: ANTHROPIC_MODEL=claude-sonnet-4-6 → proxy rewrites to glm-5.2 on the wire

The proxy applies five interventions, in order:

1
Tool-definition sanitizer. Claude Code sends Anthropic proprietary tool types (web_search_20250305, computer_20241022, etc.) that CERIT's vLLM backend rejects with HTTP 400. The proxy converts web_search_* to a standard tool definition and strips computer_* / bash_* / text_editor_* variants entirely.
2
Agentic continuation injection. For every tool-using request, the proxy appends an imperative block to the system prompt: "Call a tool immediately. Do not explain. Do not narrate." — with a BAD/GOOD example. Research on RLHF-trained models shows direct imperatives outperform polite phrasings by ~30% for compliance. This alone reduces idle-stop from 33% to 0%.
3
Turn-count and repetition guards. The proxy reads the messages array in each request (stateless — no session tracking needed) to count assistant turns and detect repeated tool calls. At turn 12: a gentle synthesis nudge. At turn 20: an imperative hard stop. If any single tool appears ≥ 3 times in the last 30 messages: a loop-break injection. This eliminated the 143-tool GLM runaway.
4
GLM thinking disable. GLM-5.2 has a reasoning ("thinking") mode that is on by default. With thinking enabled, each tool call is preceded by a long internal monologue — 143 calls × ~6 s thinking = 858 s before any output. The proxy injects chat_template_kwargs: {"enable_thinking": false} when routing to glm-5.2. T1 wall time dropped from 900 s (timeout) to 48 s.
5
Model routing and fallback chain. The incoming claude-sonnet-4-6 model name is rewritten to the CERIT target. Default for tool sessions: glm-5.2. On HTTP 400 context-overflow: retry with kimi, then qwen3.5-122b. The claude-* spoofed model name is preserved toward Claude Code, so the TUI spinner stays alive.
Why not tool_choice: any? The obvious fix for idle-stop is to force tool_choice: {"type": "any"} so the model must call a tool. This was tried and removed. kimi-k2.6's thinking phase without tool_choice:any takes 500+ s; with it, the model enters a cat-loop — repeating cat file.pbs 24+ times when it wants to write a final answer. The imperative system-prompt injection is both more reliable and doesn't prevent clean exits.

Benchmark: seven real tasks, four model configurations

All benchmarks run Claude Code via the proxy against a fixed set of seven tasks drawn from real research-engineering work on the A2507 protein electric-field project. Tasks range from a 600-file tools catalogue (T1, hardest) to a single-script write-and-execute (T6, simplest). Each run is fully automated: run_tests.py spawns Claude Code as a subprocess, captures --stream-json events, and extracts turns, tool calls, idle-stop rate, and context growth.

The GLM-5.2 (thinking OFF) column is striking: not only is it the fastest (494 s vs. qwen's 883 s, a 44% reduction), it also required zero proxy continuation injections. The model naturally called tools and exited cleanly without any prompting beyond the task description. The 0 injections figure suggests that GLM-5.2 in non-thinking mode has better intrinsic instruction following for agentic tasks than the other models tested.

DeepSeek V4 Pro deserves a note: it is fast on simple tasks but generated 68 upstream API errors (CERIT vLLM under load) and triggered auto-compact twice on T1 and T3, causing context to grow 2.4× beyond the 110 K target window. Usable, but less reliable for sustained work.

Important caveat: GLM-5.2 is on CERIT's experimental tier (qwen3.5-122b is guaranteed). The proxy fallback chain is glm-5.2 → kimi → qwen3.5-122b, so sessions degrade gracefully if GLM is offline. Check llm.ai.e-infra.cz/status/ for current availability.

Extended benchmark: 117 tasks, quality scoring (June 2026)

After the 7-task comparison above, the benchmark suite was expanded to 117 tests across 11 categories (A–K), covering every major Claude Code tool type: file reads, code generation, edit/refactor, bash scripts, search/audit, multi-tool workflows, long-context stress, vibe-style variants (terse, vague, conversational, aggressive), verification, web research, and autonomous tasks. A secondary LLM quality judge (qwen3.5-122b, 1–10 scoring) evaluated each output independently of the success regex.

Setup: step by step

Prerequisites

You need: (a) a MetaCentrum or MUNI account with access to the e-INFRA CZ LLM gateway, (b) Claude Code installed (npm i -g @anthropic-ai/claude-code or the official installer), (c) Python 3.10+ on the machine that will run the proxy.

1
Get your CERIT API token. Log in to llm.ai.e-infra.cz with your institutional account. Under your profile, generate an API key. Store it: 
mkdir -p ~/.config/cerit
echo 'YOUR_TOKEN_HERE' > ~/.config/cerit/token
chmod 600 ~/.config/cerit/token
2
Clone the setup repo. 
git clone https://github.com/mcer33/claude-code-cerit-llm.git ~/dev/claude-code-cerit-llm
cd ~/dev/claude-code-cerit-llm
bash install.sh
The install script copies cerit-rewrite-proxy.py to ~/dev/, appends the shell functions to ~/.bashrc, and optionally sets up the MCP delegation venv for subagent workloads.
3
Reload your shell and verify. 
source ~/.bashrc
claude-cerit-ping
Expected output: PONG followed by HTTP 200 | ttfb ~0.5s. If you see 401, your token file is wrong or empty.
4
Launch Claude Code via CERIT. 
claude-cerit-rich
This starts the proxy on :9999 (if not already running), sets ANTHROPIC_BASE_URL=http://127.0.0.1:9999/, and launches Claude Code with a spoofed ANTHROPIC_MODEL=claude-sonnet-4-6 so the TUI spinner stays active. On the wire the proxy routes to glm-5.2 with thinking disabled.

Proxy architecture in detail

The proxy is a single-file ThreadingHTTPServer with no dependencies beyond the Python standard library. It is intentionally simple: no async, no session state, no database. The turn-count guard works by counting "role": "assistant" entries in the messages array of each incoming request — entirely stateless. 

# Five intervention stages, applied per-request:

# 1. Tool-definition sanitizer
body_obj = sanitize_tool_definitions(body_obj)   # strips/converts proprietary types
body_obj = sanitize_tool_inputs(body_obj)         # strips extra quotes (gemma4 bug)

# 2. Agentic continuation + guards (tool-using requests only)
if request_has_tools:
    n_asst = sum(1 for m in messages if m.get("role") == "assistant")
    # repetition detection …
    injection = CERIT_CONTINUATION + turn_warn + repeat_guard

# 3. Model routing
target = MODEL_OVERRIDES.get(model) or DEFAULT_TOOL_TARGET  # → glm-5.2

# 4. GLM thinking disable
if target == "glm-5.2":
    send_obj["chat_template_kwargs"] = {"enable_thinking": False}

# 5. Context-overflow fallback chain on HTTP 400
# glm-5.2 → kimi → qwen3.5-122b

What's next

The proxy now includes a TASK_COMPLETE sentinel: the continuation rule instructs the model to end its final response with the line TASK_COMPLETE. The test runner detects this as a supplemental completion signal, disambiguating genuine completion from success-regex mismatch.

Llama-4-scout tool support remains unverified. A dedicated L01 test exercises claude-cerit-long with a minimal file-read task to confirm whether the 270 K context model supports tool calling. Until confirmed, it remains in the text-only routing tier.

DeepSeek V4 Pro (1 M token context) is worth re-testing as CERIT's vLLM infrastructure stabilises — the 68 upstream errors in run7 were likely transient load issues rather than a fundamental limitation.

The benchmark suite (run_tests.py) now covers 117 tests with --suite base|ext|all, --category A,B,..., --timeout-scale for overnight runs, and a built-in LLM quality judge. An idle-stop reproducer (cerit_idle_stop_reproducer.py) quantifies baseline vs. mitigated idle-stop rates for comparison with future CERIT server-side fixes.
